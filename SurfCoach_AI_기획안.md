# 🏄 SurfCoach AI — 기획안

## 1. 서비스 개요

| 항목 | 내용 |
|------|------|
| **서비스명** | SurfCoach AI |
| **한 줄 정의** | AI가 서핑 자세를 분석하고, 전문가 수준의 피드백을 제공하는 코칭 앱 |
| **타겟 유저** | 입문~중급 서퍼 (독학러, 레슨비 부담층) |
| **핵심 가치** | 강사 없이도 자세 교정 가능 → 레슨 비용 절감 |

---

## 2. 핵심 기능 정의

### 🎯 MVP (꼭 있어야 할 기능)

```
① 영상 업로드 → 자세 분석
   - 사용자가 서핑 영상 촬영 후 업로드
   - YOLO로 신체 키포인트 감지
   - 팝업 자세, 패들링, 스탠스 분석

② 자세 점수 + 피드백 리포트
   - 각 동작별 점수 (0~100)
   - 잘못된 자세 → 텍스트/이미지로 교정 가이드

③ 개인 히스토리 관리
   - 날짜별 분석 기록 저장
   - 성장 그래프 시각화
```

### 🚀 고도화 기능 (이후 단계)

```
④ 음성 피드백
   - 분석 결과를 TTS로 음성 안내
   - 영상 보면서 귀로 피드백 수신

⑤ 실시간 분석 (라이브 모드)
   - 해변에서 스마트폰 거치 후 실시간 코칭

⑥ 위험 자세 경보
   - 낙수 위험 자세 감지 → 안전 경고
   - (구조대 경력 활용 포인트!)
```

## 3. 서핑 자세 분석 기준 — 도메인 지식 적용

> 이 부분이 경쟁 서비스와의 진짜 차별점

### 분석할 핵심 동작 3가지

#### ① 팝업 (Pop-up)
```
체크포인트:
- 손 위치 (가슴 옆인지 여부)
- 무릎 굽힘 각도 (90~120도 이상적)
- 일어서는 속도 (프레임 수 측정)
- 시선 방향 (앞을 보는지)
```

#### ② 스탠스 (Stance)
```
체크포인트:
- 발 간격 (어깨너비 기준)
- 무게중심 위치 (앞발 60%, 뒷발 40%)
- 어깨 방향 (파도 진행방향과 각도)
- 팔 밸런스 포지션
```

#### ③ 패들링 (Paddling)
```
체크포인트:
- 입수 각도 (손끝부터 45도)
- 스트로크 깊이
- 몸통 위치 (보드 중심)
- 좌우 균형
```

## 4. 기술 스택

### 전체 아키텍처

```
[사용자 스마트폰]
      ↓ 영상 업로드
[Claude 앱 (프론트엔드)]
      ↓
[백엔드 서버]
      ↓           ↓
[YOLO 분석]   [Gemma4 피드백 생성]
      ↓           ↓
    골격 좌표  →  자연어 교정 가이드
      ↓
[음성 모델 TTS]
      ↓
[사용자에게 결과 전달]
```

### 상세 기술 스택표

| 파트 | 기술 | 역할 | 난이도 |
|------|------|------|--------|
| **프론트엔드** | React Native | iOS/Android 앱 | ⭐⭐ |
| **백엔드** | FastAPI (Python) | API 서버 | ⭐⭐ |
| **자세 감지** | YOLOv8-pose | 신체 키포인트 추출 | ⭐⭐⭐ |
| **자세 분석 로직** | Python + NumPy | 각도/거리 계산 | ⭐⭐ |
| **피드백 생성** | Gemma4 (로컬) | 자연어 교정 문장 | ⭐⭐ |
| **음성 출력** | Whisper / gTTS | TTS 음성 피드백 | ⭐ |
| **데이터베이스** | SQLite → PostgreSQL | 분석 기록 저장 | ⭐⭐ |
| **영상 처리** | OpenCV | 프레임 추출, 오버레이 | ⭐⭐ |
| **클라우드** | AWS or GCP | 배포 환경 | ⭐⭐⭐ |

## 5. 개발 로드맵

```
📍 1단계 (1~2주) - 핵심 검증
   └─ YOLOv8-pose로 서핑 영상에서 골격 추출 테스트
   └─ 각도 계산 로직 작성 (무릎각도, 어깨각도 등)
   └─ 결과값이 실제 자세와 맞는지 → 직접 검증 (강사 경력 활용!)

📍 2단계 (3~4주) - 백엔드 구성
   └─ FastAPI 서버 구축
   └─ 영상 업로드 → 분석 → 결과 반환 API 완성
   └─ Gemma4 연동 → 분석값을 자연어 피드백으로 변환

📍 3단계 (5~6주) - 프론트엔드
   └─ React Native 앱 UI 제작
   └─ 영상 업로드, 결과 화면, 히스토리 화면
   └─ Claude 앱으로 배포

📍 4단계 (7~8주) - 고도화
   └─ 음성 피드백 추가
   └─ 실제 서퍼에게 베타 테스트
   └─ 피드백 반영 후 정식 출시
```

## 7. LLM 성능 테스트 전략

### 테스트 순서 (Top-down 방식)

```
① Claude API / GPT-4o   → 기준점(baseline) 확보
         ↓
② Gemini 1.5 Pro/Flash  → 중간 성능 확인
         ↓
③ Gemma4 (로컬/무료)    → 최종 목표
         ↓
   고성능 대비 품질 차이 측정 → 배포 모델 확정
```

### 테스트 설계

```
[입력값 동일하게 고정]
- 서핑 자세 분석 결과 예시
  예) 무릎각도: 145도, 발간격: 어깨의 1.8배, 시선: 하향

[각 모델에 동일 프롬프트 입력 후 결과 저장]

[비교 평가 항목]
  - 교정 가이드의 구체성
  - 서핑 전문 용어 정확성  ← 강사 경력으로 직접 채점!
  - 초보자 이해 가능성
  - 응답 속도
  - API 비용
```

### 평가 기준표

| 모델 | 비용 | 속도 | 품질 | 비고 |
|------|------|------|------|------|
| Claude API | 유료 | 빠름 | 기준점 | baseline |
| GPT-4o | 유료 | 빠름 | 기준점 | baseline |
| Gemini Flash | 저렴 | 매우 빠름 | 중간 | 중간 검증 |
| Gemma4 | 무료 | 환경따라 다름 | 목표 | 최종 배포 목표 |

## 8. LLM 모델 추상화 설계

> 개발 버전에서는 모델을 쉽게 교체하고,
> 배포 버전에서는 최종 선정 모델만 고정해서 사용

### 핵심 아이디어 — 모델 추상화 패턴

```
[개발 버전]                    [배포 버전]
환경변수로 모델 선택            모델 고정 (Gemma4)
LLM_PROVIDER=claude      →    LLM_PROVIDER=gemma4
LLM_PROVIDER=gpt4o            변경 불가
LLM_PROVIDER=gemma4
```

### 폴더 구조 설계

```
backend/
└── llm/
    ├── base.py          ← 공통 인터페이스 정의 (뼈대)
    ├── claude.py        ← Claude API 연동
    ├── gpt4o.py         ← GPT-4o API 연동
    ├── gemini.py        ← Gemini API 연동
    ├── gemma4.py        ← Gemma4 로컬 연동
    └── factory.py       ← 환경변수 보고 모델 선택
```

### 핵심 코드 구조 (개념)

```python
# base.py - 모든 모델이 따라야 할 공통 틀
class BaseLLM:
    def get_feedback(self, pose_data: dict) -> str:
        raise NotImplementedError

# claude.py - Claude 버전
class ClaudeLLM(BaseLLM):
    def get_feedback(self, pose_data: dict) -> str:
        # Claude API 호출
        ...

# gemma4.py - 최종 배포 버전
class Gemma4LLM(BaseLLM):
    def get_feedback(self, pose_data: dict) -> str:
        # Gemma4 로컬 호출
        ...

# factory.py - 환경변수 보고 자동 선택
def get_llm_model():
    provider = os.getenv("LLM_PROVIDER", "gemma4")  # 기본값: gemma4
    if provider == "claude":
        return ClaudeLLM()
    elif provider == "gpt4o":
        return GPT4oLLM()
    elif provider == "gemma4":
        return Gemma4LLM()
```

### 환경변수 파일 (.env)

```bash
# 개발 버전 - 테스트할 모델로 변경
LLM_PROVIDER=claude        # claude / gpt4o / gemini / gemma4

# 배포 버전 - 고정
LLM_PROVIDER=gemma4
```

### 개발 vs 배포 전환 흐름

```
개발 단계
  .env → LLM_PROVIDER=claude  →  Claude로 테스트
  .env → LLM_PROVIDER=gpt4o   →  GPT-4o로 테스트
  .env → LLM_PROVIDER=gemma4  →  Gemma4로 테스트
              ↓
        성능 비교 완료
              ↓
배포 단계
  .env → LLM_PROVIDER=gemma4  →  고정 (코드 변경 없음!)
```

> ✅ 핵심 장점: 모델이 바뀌어도 나머지 코드는 전혀 수정할 필요 없음
